<objective>
Deliver the theme slice end to end: register a dark Vuetify theme alongside the existing
light one, bind Vuetify's active theme to the persisted `theme` preference from plan 04-01,
and add a light/dark toggle in Settings. After this plan a user flips the theme in Settings
and the whole UI (app bar, drawer, cards, all pages) switches live, and the choice survives
a reload because it is the same persisted preferences-store value.

preferences store (from 04-01) owns the chosen value, VueUse `useLocalStorage` (already wired
in 04-01) persists it. This plan ADDS only the theme control + the Vuetify binding; it does
not touch the unit slice or introduce new state (the `theme` field already exists in the
preferences store).

Output: a dark theme in `src/plugins/vuetify.ts`, a `useThemePreference` composable that
syncs store <-> Vuetify, the theme toggle wired into the Settings "Theme" section placeholder
left by 04-01, the binding mounted at app root, and an implementation-notes entry.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted theme preference -> Vuetify theme API | The `theme` value (originating from localStorage via 04-01) is used to select Vuetify's active theme name |
| localStorage -> app state | (Carried from 04-01) the preferences object hydrating the store is tamper-controllable |

Out of scope (no surface): authentication, authorization, sessions, secrets, server-side
state - frontend-only, no backend, no API key.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | persisted `theme` used to pick a Vuetify theme name | mitigate | The store already sanitizes `theme` against `THEME_MODES` on read-back (04-01, T-04-01); the binding only ever passes a validated `'light'`/`'dark'` value to Vuetify, so a tampered theme value falls back to the default rather than selecting a non-existent/unsafe theme. |
| T-04-06 | Spoofing/XSS | Settings/app-bar controls rendering | mitigate | The theme controls render only static labels and bind to typed store values via `{{ }}` / props; no `v-html` is introduced, so no untrusted markup is rendered. |

Block-on: high severity only. No high-severity threats - this slice only maps a validated
enum value to a Vuetify theme name; no new external input, no backend, no secrets.
</threat_model>
