<objective>
Deliver Phase 3 as one vertical slice: clicking a city on the dashboard opens its detail
page at `/city/:id`; the detail page reads the city from its route param, fetches a
multi-day forecast from Open-Meteo (axios + Vue Query), and shows both a per-day forecast
list and a Chart.js temperature chart that reacts to the selected city.

Purpose: extend the Phase 2 backbone (axios client, Vue Query, cities store, WMO util)
with a forecast path and the chart library. Each library keeps one obvious, visible job:
vue-chartjs/Chart.js = the temperature chart, Vue Query = forecast caching/loading/error,
Vue Router params = which city is shown. Readability over cleverness (study artifact).

Output: `fetchForecast` added to the axios client, a `DailyForecast` type, a
`useForecast` Vue Query composable, `ForecastList.vue` + `ForecastChart.vue` components,
a filled-in `CityDetailPage.vue`, a clickable link from each `WeatherCard`, an updated
`AppShell` City Detail link, a component test, and an implementation-notes entry.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| route param -> app logic | The `:id` from the URL (user-controllable, e.g. deep link) is used to look up a saved city and is rendered/echoed on the page |
| route param / city coords -> outbound HTTP | The selected city's latitude/longitude flow into the Open-Meteo forecast request |
| Open-Meteo forecast response -> DOM + chart | Untrusted external JSON (dates, temperatures, weather codes) is rendered in the list and fed into Chart.js |

Out of scope (no surface in this app): authentication, authorization, session/credential
management, and secret handling - there is no backend, no login, no API key, and no
server-side state. Classic STRIDE Spoofing/Repudiation/Elevation-of-privilege threats do
not apply to a static, no-auth, public-API frontend and are intentionally not enumerated.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | `:id` route param used to resolve a city | mitigate | The param is only used to look up an existing in-memory SavedCity by matching `String(city.id)`/`city.key`; an unrecognized/crafted id resolves to "no city" and renders the friendly not-found state - it is never used to build a request directly. |
| T-03-02 | Spoofing/XSS | rendering route param + API strings (city name, dates, condition labels) in the detail page and list | mitigate | All dynamic strings are rendered via Vue text interpolation `{{ }}` (auto-escaped); `v-html` is NOT used anywhere in the new components, so injected markup cannot execute. |
| T-03-03 | Tampering | forecast request params (lat/lon/days) | mitigate | Coordinates come from the typed `SavedCity` (numbers), passed via axios `params` (URL-encoded) on the shared `http` client - never string-concatenated into the URL. |
| T-03-04 | Denial of Service | malformed / oversized / mismatched-length forecast arrays breaking the chart or list | mitigate | `fetchForecast` maps to a fixed `DailyForecast` shape; the list/chart iterate by index over the returned arrays. Render the forecast only on query success (guarded by `isPending`/`isError`), so a failed/empty response shows the error/empty state rather than throwing; the chart is mounted only when `data` exists. |
| T-03-05 | Information Disclosure | error surfaces on the detail page | accept/mitigate | Public no-key API: no secrets exist to leak. Errors are mapped to a generic inline message; the raw axios error object/stack is never rendered (mitigates accidental detail exposure). |
| T-03-06 | Denial of Service | rapid city switching -> overlapping forecast fetches | mitigate | `useForecast` is keyed by `city.key` and forwards the Vue Query `signal`, so superseded requests abort and Vue Query dedupes/caches per city. |
| T-03-SC | Tampering | npm installs (chart.js, vue-chartjs) | mitigate | Both are mainstream, widely-used, pre-agreed packages on the project's chart stack (chart.js ^4.5.1 verified on registry.npmjs.org; vue-chartjs ^5.3.3 verified, peer chart.js ^4). No `[ASSUMED]`/`[SUS]`/`[SLOP]` package introduced; no other dependency added. No blocking legitimacy checkpoint required. |

Block-on: high severity only. No high-severity threats in this frontend-only, no-backend,
no-auth, no-key slice.
</threat_model>
