<objective>
Build the user-facing slice on top of plan 02-01's backbone: a debounced, validated
city-search autocomplete; a weather card that shows loading/error/content for each saved
city; and the DashboardPage that assembles search + cards + empty-state prompt. After
this plan a user can search a city and watch its current weather appear, end to end.

Vue Query → UI). Each library keeps one visible job: vee-validate+yup validate the
search, Pinia holds saved cities, Vue Query drives per-card loading/error/cache, the WMO
util turns codes into label+icon.

Output: `src/components/CitySearch.vue`, `src/components/WeatherCard.vue`, and a filled-in
`src/pages/DashboardPage.vue`.
</objective>

    SRCH-02, SRCH-03, D-01, D-02, D-05).

    Search UI (D-01): a Vuetify `v-autocomplete` whose items are geocoding candidates.
    Bind `:items` to a reactive list of `GeoCity`, set `:item-title` to a function/format
    showing name + admin1 + country (so ambiguous names like "Paris" are distinguishable),
    and `:item-value` to the whole item (return-object) so selection gives a full
    `GeoCity`. Drive item loading from the typed search term, not from the v-model
    selection: use `:loading` while a geocode request is in flight and `no-filter` so
    Vuetify does not re-filter server results.

    Debounced geocoding (D-01): watch the typed search text and debounce it (~300ms;
    implement a small inline debounce with `setTimeout`/clearTimeout — do NOT add a new
    dependency such as lodash). On each debounced change, only call `geocodeCity` if
    validation passes (see below). Forward an `AbortController` signal and abort the
    previous request when a new keystroke supersedes it. Store results into the items ref;
    on an empty result the autocomplete simply shows no matches.

    Validation (D-02, SRCH-02): use vee-validate + yup on the typed search string. Define
    a yup string schema: `required` (non-empty) and `min(2)` / `max(80)` (sensible
    length). Use vee-validate (`useField` or a `<Form>`/`useForm` pattern) to expose an
    error message and to gate geocoding — do not call `geocodeCity` while the current term
    is invalid. Surface the validation message via the `v-autocomplete`'s
    `:error-messages`.

    Selection → store (D-05): on `update:modelValue` / selection of a `GeoCity`, call
    `useCitiesStore().addCity(geoCity)` (the store handles dedupe — a repeat city is a
    no-op, so no duplicate card). After adding, clear the selection/search so the field is
    ready for the next city.

    Keep it readable for a learner: short comments naming each library's job
    (vee-validate = validation, geocodeCity = HTTP search, store.addCity = saved cities).
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → search/geocoding | Free-text city name from the autocomplete drives an outbound request |
| Open-Meteo response → DOM | External JSON strings (city name, country, condition label) are rendered in cards |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Tampering | search input length/shape | mitigate | vee-validate + yup enforce non-empty + min(2)/max(80) before any `geocodeCity` call; oversized/empty queries never reach the network. |
| T-02-05 | Spoofing/XSS | rendering API strings in cards | mitigate | All external strings (name, country, WMO label) are rendered via Vue text interpolation `{{ }}`, never `v-html`; Vue auto-escapes, so injected markup cannot execute. |
| T-02-06 | Information Disclosure | error display | mitigate | Cards render only generic mapped messages ("City not found" / "Could not load weather"); the raw axios error object/stack is never shown to the user. |
| T-02-07 | Denial of Service | rapid typing → request storm | mitigate | ~300ms debounce + AbortController cancels superseded geocode requests; Vue Query dedupes/caches per-card weather fetches. |

Block-on: high severity only. No high-severity threats — frontend-only, no backend, no
auth, no secrets, no `v-html`.
</threat_model>
