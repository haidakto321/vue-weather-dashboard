<objective>
Build the preferences backbone and the first user-visible preference slice end to end: a
persisted preferences Pinia store (temperature unit, theme, language), VueUse-backed
localStorage persistence for both preferences AND the existing saved-cities store, and a
working temperature-unit toggle in Settings that updates every temperature on screen
(cards, forecast list, chart) live and survives a reload.

Purpose: establish the single source of preference state that plans 04-02 (theme) and
04-03 (language) extend, while delivering one complete vertical slice now (unit toggle ->
live UI update -> persistence). Each library keeps one obvious, visible job: Pinia = the
preference + cities client state, VueUse `useLocalStorage` = persistence (CMPS-01), a small
composable = unit conversion. The `theme` and `language` fields are defined in the store
here (so 04-02/04-03 only add their controls + consumers, not new state) but the unit
toggle is the only control wired in this plan.

Output: `src/types/preferences.ts`, `src/stores/preferences.ts`, a persisted
`src/stores/cities.ts`, `src/composables/useTemperature.ts`, unit-aware
WeatherCard/ForecastList/ForecastChart, a Settings unit control, two unit tests (store +
composable), and an implementation-notes entry. `@vueuse/core` and `vue-i18n` are installed
here so plan 04-03 needs no install step.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage -> app state | Values read back from `weather-prefs` and `weather-cities` are user/tamper-controllable persistent storage that hydrate Pinia stores and then drive Vuetify theme, i18n locale, and the card UI |
| Open-Meteo response -> DOM | (Carried from Phase 2/3) external JSON strings (city name, country) still flow into cards/list rendered this phase |

Out of scope (no surface): authentication, authorization, sessions, secrets, server-side
state - frontend-only, no backend, public no-key API.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | preferences read back from `weather-prefs` localStorage | mitigate | The preferences store sanitizes every field on read-back against `TEMPERATURE_UNITS`/`THEME_MODES`/`LANGUAGES`; an unknown/corrupt value falls back to `DEFAULT_PREFERENCES`, so a tampered entry can never feed an invalid unit/theme/locale into the UI or crash hydration. |
| T-04-02 | Tampering | saved cities read back from `weather-cities` localStorage | mitigate | On read-back the cities store validates each entry is an object with a string `key` and numeric `id`/`latitude`/`longitude`, dropping malformed entries, so a tampered array cannot inject malformed objects into the cards/queries. |
| T-04-03 | Spoofing/XSS | rendering stored city names / preference-derived strings in the DOM | mitigate | All dynamic strings (city name, country, unit symbol) are rendered via Vue text interpolation `{{ }}` (auto-escaped); no `v-html` is introduced in any file this plan touches. |
| T-04-04 | Denial of Service | malformed JSON in a localStorage key breaking JSON.parse at startup | mitigate | VueUse `useLocalStorage` catches parse errors and returns the provided default; combined with the sanitize step the app boots to safe defaults rather than throwing on corrupt storage. |
| T-04-SC | Tampering | npm installs (@vueuse/core, vue-i18n) | mitigate | Both are mainstream, widely-used, pre-agreed Phase-4 packages named in REQUIREMENTS.md (VueUse for CMPS-01/PERS-01, vue-i18n for I18N-01/02); verified on registry.npmjs.org (@vueuse/core, vue-i18n). No `[ASSUMED]`/`[SUS]`/`[SLOP]` package introduced; no other dependency added. No blocking legitimacy checkpoint required. |

Block-on: high severity only. No high-severity threats in this frontend-only, no-backend,
no-auth, no-key slice.
</threat_model>
