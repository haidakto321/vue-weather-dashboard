<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| persisted language preference -> vue-i18n locale | The `language` value (from localStorage via 04-01) selects vue-i18n's active locale |
| i18n message values + interpolated data -> DOM | Translated strings (static, app-authored) and any interpolated runtime values (e.g. city name) are rendered as UI text |

Out of scope (no surface): authentication, authorization, sessions, secrets, server-side
state - frontend-only, no backend, no API key.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Tampering | persisted `language` used to pick the i18n locale | mitigate | The store sanitizes `language` against `LANGUAGES` on read-back (04-01, T-04-01) and i18n uses `fallbackLocale: 'en'`; a tampered/unknown locale value falls back to a registered locale rather than breaking translation. |
| T-04-08 | Spoofing/XSS | rendering i18n strings (incl. any interpolated city name) in the DOM | mitigate | vue-i18n `t()` output is rendered via Vue text interpolation `{{ }}` (auto-escaped); no `v-html`/`v-t` raw-HTML rendering is used, and message catalogues are app-authored static strings - no untrusted markup can execute. Any interpolated runtime value (city name) is passed as an interpolation arg, still text-escaped. |
| T-04-09 | Denial of Service | missing translation key throwing/breaking render | mitigate | en/ja catalogues are kept in key parity and `fallbackLocale: 'en'` is set, so a missing key degrades to the English string (or the key name) rather than crashing the view. |

Block-on: high severity only. No high-severity threats - app-authored static messages, a
validated locale enum, no backend, no secrets, no raw-HTML rendering.
</threat_model>
